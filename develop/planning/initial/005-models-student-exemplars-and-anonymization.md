# Models, Student Exemplars, and Anonymization

## Role of models in visual learning

Part of the visual learning framework involves showing learners what successful work looks like. This may include a teacher-created or otherwise appropriate model that demonstrates the expected form, quality, or outcome.

After an assignment has been taught at least once, authentic student models may also become useful. These can provide realistic examples of successful work produced under the same classroom conditions.

The exact requirements for selecting and presenting models should ultimately follow the authoritative visual learning documents and procedures when they are added.

## Student models

Student work may be included as an exemplar only when it has been anonymized.

At minimum, a student model should not expose:

- The student's name
- A username or email address
- A student identification number
- Teacher comments that identify the student
- File metadata or visible document properties that identify the student
- Names of classmates or other individuals appearing in the work
- Other contextual details that make the student readily identifiable

Removing the visible student name is necessary but may not always be sufficient.

## Required preflight check

Even when the teacher has already reviewed and anonymized a student model, the system should provide an additional check before the material is stored, processed, displayed, or shared.

The check should look for likely identifying information and clearly distinguish between:

- No apparent identifiers found
- Possible identifiers requiring teacher review
- Definite identifiers that must be removed before use

The teacher should remain the final reviewer. An automated check should be treated as a safeguard rather than a guarantee that anonymization is complete.

## Handling and access boundary

Student exemplars introduce a different privacy category from teacher-authored assignments and public-domain or licensed source material.

The eventual system should make deliberate decisions about:

- Whether anonymized student models may be retained
- Whether they may be included in a repository
- Whether they may be displayed through the interface
- Whether access should remain limited to the teacher
- Whether models should be stored separately from other assignment materials
- Whether originals should be deleted after an anonymized version is created

A student model should not be assumed safe for public exposure merely because a name has been removed.

## Relationship to assignment processing

The system may eventually help with tasks such as:

- Associating one or more models with an assignment
- Distinguishing teacher-created models from student exemplars
- Checking submitted exemplars for identifiers
- Comparing an assignment's directions and rubric with the selected model
- Confirming that the model actually demonstrates the intended success criteria
- Preparing a student-facing version that does not reveal protected information
- Recording which assignments already have an appropriate model and which still need one

These capabilities should support the teacher's established visual learning procedure rather than define that procedure independently.

## Open questions

- Is teacher or student consent required before retaining an anonymized exemplar?
- Should student models ever be accessible outside the teacher's authenticated workspace?
- Should the system preserve the original file, or only an approved anonymized copy?
- How should handwritten names, embedded images, comments, revision history, and document metadata be checked?
- Should a model be linked to a particular assignment version, rubric version, course, or school year?
- What criteria determine whether a student model is strong enough and appropriate enough to use?
